# 项目演进与未来规划 (Roadmap)

`FE-Auto-Factory` 目前已经正式演化到 `v3.3.0` 大规模稳定星环内核纪元（剥离驱动沙盒与 EJS，并且能完美实现与用户手写代码逻辑共生的 AST 挂载合并算法）。

但是，这仅仅是我们定义“让业务系统光速生根发芽”的起点。我们的愿景是最终从一条 Schema 的声明链中连接大模型的推断，直到让大厂和业务前台再也没有重构“造轮子”的必要。

以下是我们的远景与 Roadmap 规划：

## 近期计划 (Coming Soon)

1. **可视化 Schema 编辑生态编辑器 (IDE Plugin & Web-Portal)**
   - 我们现在完全依赖 `yaml`/`json` 手写及通过 Swagger `sync` 自动推导。这仍然有一丝“工程师属性”的壁垒。
   - 未来我们将直接抛出 Web 端拖拽和配置系统。通过拖拽一个“表格组件”或者给它绑定上一个微服务 API 接口名，后台通过拖拽数据自动实时为您吐出工厂的标准 `Schema.yaml`！
   
2. **全栈向生成 (BFF 生成下放)**
   - 目前我们的架构聚焦于前端三驾马车：视图 (UI)、逻辑 (Hooks/Store) 和 胶水层 (API Services / Playwright E2E)。
   - 在下一步 `preset` 插件机制扩展里，我们将让插件接管 BFF (Backend For Frontend) —— 比如你选定了 `react-nest` 架构引擎包。我们在生成客户端骨架的同一秒，利用 Schema 将为您生成 `NestJS` 或 `NextJS API Routes` 下的中转路由或 Controller 模型，实现一次配置，全栈打通。

## 中期规划 (Mid-Term)

3. **Schema 版本留痕与 Diff（云端快照挂载）**
   - 打破单机项目的目录壁垒，所有 `page.schema.yaml` 被推向一个中枢系统进行管理和版本发布（就像现在公司用以沉淀数据的 Rap 或者 YAPI 平台一样）。
   - 实现**前端即服务 (FaaS for UI)**：不仅只是生成代码到本地 Git，而是当修改远端图纸版本后，引擎通过 WebHooks 等触发机器进行二次无损 AST 渲染合并，自动在 CI 里将最新生成的业务特性发布为可用。

4. **异常埋点自愈循环 (Self-healing Loop)**
   - 与 `04-deployment` 后末端做深化对接。如果我们在生产环境中收到了大量的由于生成视图里缺漏了一个异常状态处理导致的 Sentry 报警；AI 故障自愈系统捕捉到特定的 `telemetry` track-id 和抛锚上下文。
   - 它可以智能下发 PR (Pull Request) 给开发者，内容是：“引擎探测到此页面的某某请求时常超时，已为您更新了该部分的 Schema 的 `retry_strategy: true` 开关；并利用微内核将 Hook 处的相关片段进行了更新替换。”

## 长期愿景 (Long-Term)

最后，正如工程的定义，我们希望此系统消亡。当微前端化和智能化工具高度渗透以后，绝大多数 B 端和工具类的项目根本无需手动管理和提交这一个个复杂的工厂项目。前端体系会如同流体一样成为无形的底层组装系统。我们将只是在写一篇图文并茂的运营策划书时，你的代码就如同“编译后”的内容向你的受众展现了。
